## 1. 调研结论：业界蜂群 / 团队模式

多 agent 协作目前已收敛为三种核心拓扑，各类框架只是在其上做封装。

| 拓扑 | 一句话 | 代表 | 适合 |
| --- | --- | --- | --- |
| **Supervisor / 编排者-工人** | 一个 lead 拆任务、派给专才 worker、汇总结果 | OpenAI Agents SDK、LangGraph、Anthropic Research | 可拆成并行子问题、需可控/可审计 |
| **Swarm / 去中心化交接** | 对等 agent 各自决定"自己处理还是交给同伴" | OpenAI Swarm（已停更转 Agents SDK）、Strands | 高韧性、水平扩展，但难调试 |
| **Hierarchical / 角色链** | Researcher→Writer→Reviewer 多层流水线 | CrewAI、Microsoft Agent Framework | 强角色分工 |

**已成共识的硬经验（直接约束我们的设计）：**

1. **从中心化起步，遇到真实瓶颈再去中心化。** Swarm 的头号故障模式是 ping-pong 交接死循环，必须配 loop 检测 + guard。
2. **Anthropic 多 agent 研究系统**是最对口的范式：lead 维护策略与状态，spawn 子 agent **并行、各自独立 context window**，把"压缩后的精华 token"回传给 lead。效果比单 agent 高约 90%，但**烧约 15× token**，且**对紧耦合的编码任务不划算**——子任务越独立越赚。
3. **子 agent 需要完整"契约"**才不漂移：目标 / 输出格式 / 可用工具与来源 / 任务边界，缺一个就跑偏。
4. **按复杂度缩放 agent 数**：简单查证 1 个、对比类 2–4 个、复杂研究 10+。否则会为小问题滥开子 agent。
5. **Claude Code subagent 模型**是最贴近 harness 的工程实现：子 agent 起一个**全新隔离 context**、看不到主线历史、用完只回传**一条 summary**；限制是**不能嵌套**、**不实时共享**、**同文件并发会冲突**（靠 worktree / 目录分片隔离）。
6. **标准化趋势**：MCP 标准化了"agent 怎么够到工具"；Google **A2A（Agent-to-Agent）协议**正在标准化"agent 之间如何编排"（用 Agent Card 描述能力）。这给"agent 无关的蜂群"提供了现成的传输层思路。

## 2. openagent 现状评估

openagent 是一个工程级的单 agent harness：streaming agent loop + 工具系统 + context 管理 + trace/eval/权限/runtime-warning，成熟且干净。把上面的范式投到代码上，会发现**关键钩子已经埋好**，编排层是"加法"而非"重构"。

| 蜂群需要的能力 | openagent 已有的钩子 | 缺口 |
| --- | --- | --- |
| 子 agent 角色标记 | `AgentConfig.mode = "primary" \| "subagent"`（`core/types.py:100`，**已预留**） | 无子 agent 运行路径 |
| 现成角色 | `UniversalAgent` / `PlanAgent` / `ExploreAgent` + 提示文件 | 无 reviewer/critic 角色、无角色注册表 |
| 只读工具裁剪 | `AgentLoop._tools_for_agent` 已把 `tools="readonly"` 映射到 `{read, glob, grep, ls, skill, todoread, question}`（`core/loop/processor.py:205`） | —（直接复用） |
| 隔离 context | 每个 agent 自带 `Session` + `ContextPack`，天然隔离 | 无 spawn / 汇聚机制 |
| 因果追踪 | `TraceEvent.span_id` / `parent_span_id` **已是分层结构** | 无人写入 agent 父子关系 |
| 每会话工作区 | `build_workspace_runtime(session)`（`core/execution/runtime.py:80`） | 无 worktree / 目录分片策略 |
| 成本/限额闸 | `runtime_warnings` 事件 + eval CI gate | 无 fan-out 级别的聚合预算 |

**唯一真正缺的接线：** `ToolContext`（`core/tool/definition.py:22`）携带 `session_id` / `session_root` / `workspace_runtime` / `extra`，但**不携带 provider / toolkit / permission 句柄**，所以工具无法构造子 `AgentLoop`。方案的核心新增，就是把一个**编排句柄注入 `ToolContext.extra`**。

> 命名注意：openagent 已有 `AgentAdapter`（model+config 的 reply-stream 适配器），因此内核的"可插拔 agent"协议改名为 **`AgentRunner`**，避免冲突。

## 3. 核心理念：独立、agent 无关的蜂群内核

不把多 agent 做进 openagent 内部，而是抽出一个**独立的蜂群内核**：

- **解耦。** 内核是独立包，**零依赖 openagent**。依赖方向严格单向：`openagent → swarm`，内核永不 import openagent。内核可在"只有远程 HTTP agent、没装 openagent"时使用；openagent 也可在"没装内核"时作为单 agent 使用。
- **协议优先。** 一个小而稳的 `AgentRunner` 协议是唯一契约，其余（拓扑、调度、trace）都建在其上。
- **传输无关。** runner 可以进程内、子进程、或网络远程执行，内核一概不关心。
- **从中心化、只读、可观测起步。** 先 Supervisor 拓扑、先只读 fan-out、先 trace lineage。去中心化与可写 agent 是后续可选阶段。
- **数据边界。** 内核 trace 默认仅元数据；提示/输出/工具 IO 内容需显式 opt-in（沿用 OA-016 规则）。

## 5. `AgentRunner` 协议（契约本体）

内核的唯一事实来源（示意）：

```python
@dataclass
class AgentSpec:                 # 交给 worker 的契约
    role: str                    # "explore" | "research" | "review" | ...
    objective: str               # worker 独占的单一产出
    context: str = ""            # 最小必要事实（worker 看不到 lead 历史）
    boundaries: str = ""         # 范围外 + 允许的工具/来源
    inputs: dict = field(default_factory=dict)            # 工件/文件/url 引用
    output_schema: dict | None = None                     # 要求的返回结构
    limits: RunLimits = field(default_factory=RunLimits)  # 步数/token/成本/超时
    permissions: str = "READONLY"

@dataclass
class AgentResult:               # 压缩的、可程序化校验的返回
    status: Literal["completed", "partial", "failed", "cancelled"]
    summary: str
    evidence: list[str] = field(default_factory=list)     # file:line / url 锚点
    open_questions: list[str] = field(default_factory=list)
    confidence: float = 0.0
    artifacts: list[ArtifactRef] = field(default_factory=list)
    usage: Usage = field(default_factory=Usage)           # token/成本/步数/延迟
    handoff: HandoffRequest | None = None                 # 去中心化拓扑用

@dataclass
class AgentDescriptor:           # 能力广告（类似 A2A Agent Card）
    id: str
    roles: list[str]
    tool_groups: list[str]
    model_tier: str              # "lead" | "worker"
    max_context: int
    supports_streaming: bool

class AgentRunHandle(Protocol):
    def events(self) -> AsyncIterator[AgentEvent]: ...    # 流式进度/trace
    async def result(self) -> AgentResult: ...
    async def cancel(self) -> None: ...

class AgentRunner(Protocol):
    @property
    def descriptor(self) -> AgentDescriptor: ...
    async def start(self, spec: AgentSpec, ctx: RunContext) -> AgentRunHandle: ...
```

`RunContext` agent 无关，携带 trace 身份与控制：`run_id`、`parent_span_id`、取消信号、预算令牌。

**四要素契约（防漂移）：** 每个 `AgentSpec` 必须带齐 `objective / output_schema / context / boundaries`，缺一即漂移。内核在派发前校验，与具体 runner 无关。

## 6. 内核组件

- **Registry**：runner 按 id 与 `AgentDescriptor` 注册；路由/调度依据描述符（角色匹配、成本档、context 大小、是否流式）。
- **Scheduler**：有界并发 + `FanoutBudget`（最大并发、单次最大 worker 数、聚合 token/成本上限）；超限以 warning 暴露，对齐 openagent 的 `runtime_warnings`。
- **Topology（可插拔策略）**——决定"派什么、如何路由结果/交接、何时停止"：
  - `Supervisor`（v0）：lead 拆解 → 派 worker → 汇总。
  - `Hierarchical`：角色链 Researcher→Builder→Reviewer。
  - `Decentralized`（可选，最后）：对等交接 + ping-pong 检测。
- **Trace**：agent 无关事件，带 `span_id` / `parent_span_id` 因果；内核拥有蜂群级 span 树，每个 runner 挂在其 spawn span 下。默认仅元数据。
- **Aggregator**：把多个 `AgentResult` 综合成最终结果；综合器本身也可以是一次 `AgentRunner` 调用（"synthesis"角色）。

## 7. runner 种类（"接入任何 agent"如何实现）

| Runner | 传输 | 用途 | trace 精度 |
| --- | --- | --- | --- |
| `OpenAgentRunner` | 进程内 | 参考实现；最低延迟、最丰富 trace | 完整 |
| `SubprocessRunner` | stdio JSON | 任何 CLI agent（`openagent run --json` 等） | 中 |
| `HttpRunner` / `A2ARunner` | HTTP / A2A | 远程 agent、跨厂商混合蜂群 | 粗 |

"openagent 只是其中一种"由此落地：`OpenAgentRunner` 是进程内参考实现，其余 agent 经子进程或 HTTP/A2A 实现同一协议接入。蜂群可以是**异构的**——一个 openagent worker、一个外部研究 agent、一个编码 agent 同处一队。

## 8. openagent 如何接入（参考适配器）

`OpenAgentRunner.start(spec, ctx)`：

1. 建一个 `Session`（分叉目录 / 共享只读工作区）；
2. 建一个 `AgentLoop`，`mode="subagent"`、`tools="readonly"`、用 lead 的模型、`READONLY` 权限；
3. 把 loop 的流式事件映射成内核 `AgentEvent`，盖上 `parent_span_id = ctx.parent_span_id`；
4. 返回 `AgentRunHandle`，完成时产出 `AgentResult`。

**lead 的 `spawn_agent` 工具不再硬编码 spawn**，而是变成调用**内核**的瘦客户端（句柄注入 `ToolContext.extra["swarm"]`——唯一新接线，向后兼容、无全局单例）。lead 把 spec 提交给内核，由内核选 runner。

## 9. 完整演进路径（Phase 0–4）

每个阶段独立交付。Phase 0–1 单进程、只读。

### Phase 0 — 协议 + 进程内 Supervisor（内核 MVP）
- `SW-001` 在独立 `swarm/` 定义 `AgentRunner / AgentSpec / AgentResult / AgentDescriptor / RunContext`（零 openagent 依赖）。
- `SW-002` `Supervisor` 拓扑 + 调度器 + `FanoutBudget` + 并发上限。
- `OA-018` `OpenAgentRunner`（进程内），基于 `AgentLoop`，`tools="readonly"`。
- `OA-019` `spawn_agent` 工具 → 经 `ToolContext.extra["swarm"]` 调内核。
- **硬依赖 `OA-002`**（context 单一组装路径）。
- **产出：** 只读并行 fan-out，无写、无持久化、无新 trace schema。

### Phase 1 — 内核加固（可观测 + 路由 + 评测）
- `SW-003` agent 无关 trace lineage + 能力描述符路由。
- `OA-020` 把 openagent trace 事件映射进内核 lineage。
- `SW-007` 拓扑无关的 LLM-as-judge + rubric 评测（多 agent 无唯一正确路径，写操作前必须有）。

### Phase 2 — 进程外 runner（"接入任何 agent"解锁点 ★）
- `SW-004` `SubprocessRunner`（CLI agent，stdio JSON）。
- `SW-005` `HttpRunner` / `A2ARunner`（远程 agent，A2A Agent Card）。
- **产出：** 非 openagent agent 进场，跨厂商团队成为可能。

### Phase 3 — 拓扑 + 异构可写团队
- `SW-006` `Hierarchical` 拓扑 + 结构化交接 schema（`summary/evidence/open_questions/confidence/tool_state`）+ ping-pong loop 检测 + reviewer/critic 角色 + 写并发隔离（worktree / 目录所有权）。

### Phase 4 — 持久团队 + 去中心化 swarm（可选）
- `SW-008` 持久化团队 `Store` 协议（内核侧）+ 可选 `Decentralized` 对等交接拓扑。
- `OA-021` openagent 用持久化 session（`OA-003`）实现内核 `Store`，支持团队状态跨会话暂停/恢复 + 共享 blackboard 记忆。

### 推荐总顺序

## 10. 打包与边界（建议，待确认）

**推荐：** 仓内独立包 `src/swarm/`，并施加**零 import 规则**（`swarm/` 下不得 `import openagent`），独立测试；待协议稳定后再抽成独立 `pyproject`/仓库。这样**立刻拿到解耦纪律、零拆库开销**，干净的 import 边界让日后抽离变成机械操作。CI 应断言 `swarm/` 无 openagent import，把边界焊死。

备选：① day-1 直接独立仓（最干净但协调成本最高）；② 先做进 openagent 核心日后再抽（最快，但耦合蔓延，违背"蜂群独立"目标）。

## 11. 已锁决策与开放问题

**已锁（2026-06-12）：**
1. 依赖方向 `openagent → swarm`，永不反向。
2. 协议名 `AgentRunner`（避开已占用的 `AgentAdapter`）。
3. 内核注入点 `ToolContext.extra["swarm"]`（向后兼容、无全局单例）。
4. v0 worker 与 lead **同模型**；成本纯靠 `FanoutBudget` 控；per-role 模型选择推迟到 Phase 3。
5. 结果用**紧凑 JSON** 回传 lead（可程序化读 `confidence`/`open_questions`，省 token）。
6. 失败语义：runner error/超时/预算耗尽 → 返回带 `status` 的 partial 结果，**绝不 raise 进 lead loop**。
7. 起步拓扑只用 `Supervisor`；去中心化是最后可选阶段。
8. 评测：拓扑无关的 LLM-as-judge + rubric（Phase 1），写操作前必备。

**开放问题：**
- 进程外传输顺序：`SubprocessRunner`（stdio，便宜好验证）先，还是 `A2ARunner`（对齐 A2A 标准）先。**当前倾向：subprocess 先，A2A 次之。**
